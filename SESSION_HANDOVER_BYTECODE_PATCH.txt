SESSION HANDOVER BYTECODE PATCH

================================================================================
PROJECT
================================================================================

Tujuan pekerjaan pada session ini adalah melakukan reverse engineering terhadap
file bytecode Lua 5.1 milik game Mobile Legends Adventure (MLA) untuk menemukan
dan mematikan fungsi pengecekan kepemilikan hero (isHeroNotOwned) pada kelas
CommonArrangeHelper. Target akhir adalah menginjeksi hero APAPUN (termasuk hero
yang tidak dimiliki player) ke dalam slot formasi battle.

Game: Mobile Legends Adventure (MLA)
Platform: Android (LDPlayer Emulator)
Teknologi: Unity + tolua++ (C++ binding untuk Lua)
File target: assets/1/189b07cc4bd84b827a5328770329e9a3.mt

================================================================================
PROBLEM
================================================================================

1. FORMAT FILE .mt.luac

File .mt adalah format encrypted milik Roo/Anti-tamper. Diawali header "Antm".
Setelah di-decode, menghasilkan file .mt.luac yang merupakan raw Lua 5.1 bytecode
dengan header standar (1B 4C 75 61 = "\x1bLua"). Ukuran decoded file target
adalah 38491 bytes. File .mt asli dari APK berukuran 10497 bytes.

2. OBFUSCATED BYTECODE

Bytecode Lua di dalam file .mt.luac sudah mengalami obfuscation berat, termasuk:

  a. Control-flow flattening menggunakan pasangan FORPREP + FORLOOP dengan
     nilai sBx = -131071 (bx = 0). Ini membuat loop body berisi seluruh
     instruction setelah FORPREP, dan nilai variabel loop mengontrol alur
     eksekusi yang sebenarnya.

  b. Instruction palsu: SETGLOBAL r0, k1024 (k1024 out of range) dan
     GETUPVAL r2, uXXXXX (XXXXX adalah nilai sangat besar/random).
     Instruksi-instruksi ini tidak pernah dieksekusi secara real dan hanya
     sebagai filler obfuscation.

  c. RETURN instruction dengan nilai return count acak digunakan sebagai
     jump target dalam control flow flattening.

3. PARSER GAGAL

Parser awal yang dibuat membaca source string sebagai 1-byte size, padahal
Lua 5.1 menggunakan sizeof(size_t) = 4 bytes untuk string size. Ini menyebabkan
semua offset fungsi header setelah source string menjadi salah.
Akibatnya: linedefined, lastlinedefined, numparams, is_vararg, maxstacksize,
dan codesize semuanya terbaca dari offset yang salah.

4. MULTIPLE SCRIPTS CONCATENATED

Satu file .mt dapat berisi multiple Lua scripts yang digabung (concatenated).
Instruction count dari main function script pertama mencakup data dari script
berikutnya. Script fix_and_decompile.py menggunakan pendekatan: find last
valid RETURN instruction, truncate di sana.

5. FORMAT size_t

Source string di Lua 5.1 bytecode menggunakan size_t (4 bytes) untuk
menyimpan panjang string, BUKAN 1 byte. Ini adalah kesalahan kritis yang
membuat seluruh offset parsing salah.

6. CONSTANT TABLE

Constant table menggunakan format: type(1 byte) + data(bergantung type).
Type 4 = string: size(4 bytes) + content(null-terminated).
Type 3 = number: 8 bytes (double).
Type 1 = boolean: 1 byte.
Type 0 = nil: no data.

7. LIBHOUDINI LIMITATION

Game berjalan di LDPlayer melalui libhoudini (ARM to x86 translation).
libagame.so adalah ARM64 library yang di-translate. Frida di sisi x86_64
tidak bisa mengakses symbol Lua API dari libagame.so karena symbol tersebut
hanya ada di ARM code. dlopen/dlsym juga tidak pernah fire untuk fungsi
dari libagame.so.

================================================================================
ROOT CAUSE
================================================================================

PARSER GAGAL KARENA:

1. Source string size dibaca sebagai 1 byte (uint8), padahal Lua 5.1
   menggunakan sizeof(size_t) = 4 bytes (int32). Ini menyebabkan:

   - Offset linedefined, lastlinedefined, numparams, is_vararg, maxstacksize,
     codesize semua bergeser 3 byte
   - codesize yang benar ada di offset file 28-31, bukan 25-28
   - Akibatnya codesize terbaca sebagai nilai raksasa (2600731136) yang
     jelas salah untuk file 38KB

2. String constant size juga menggunakan sizeof(size_t) = 4 bytes.
   Parser awal membaca size sebagai 1 byte.

3. Fungsi header memiliki field order:
   source_string(size_t_size + content) + linedefined(int) +
   lastlinedefined(int) + nups(uint8) + numparams(uint8) +
   is_vararg(uint8) + maxstacksize(uint8) + codesize(int)

   BUKAN:
   source_string(size_byte + content) + linedefined + ...

4. Line info size (unsigned int 4 bytes) + line info array (2 bytes per entry
   dalam implementasi Lua standar, tapi beberapa custom implementation
   menggunakan 4 bytes per entry). Ini menyebabkan parsing debug info gagal
   dan crash saat membaca sub-function.

5. Sub-function disimpan INLINE (bukan sebagai offset), sehingga offset
   sub-function yang dibaca dari tabel adalah absolut (dari awal chunk),
   bukan relatif. Parser yang salah membaca offset menyebabkan crash.

TEMUAN TEKNIS FORMAT YANG BENAR:

HEADER FILE (12 bytes):
  [0-3]   magic: 1B 4C 75 61 ("\x1bLua")
  [4]     version: 51 (Lua 5.1)
  [5]     format: 00 (official)
  [6]     endianness: 01 (1 = little endian)

  [12-15] source_size: 00 00 00 00 = 0 (null source, 4 bytes size_t)
          -> BUKAN [12] = 00 (1 byte) seperti yg diasumsikan sebelumnya
  [16-19] linedefined: 00 00 00 00 = 0 (4 bytes int)
  [20-23] lastlinedefined: 00 00 00 00 = 0 (4 bytes int)
  [24]    nups: 00 = 0 (1 byte)

Ini sudah diverifikasi dengan unluac --disassemble yang berhasil mem-parse
file dan menghasilkan output yang cocok.

11. BYTE CODE ARCHITECTURE

Struktur instruction Lua 5.1:

  iABC format:
    opcode(6 bit) | A(8 bit) | C(9 bit) | B(9 bit)
    Untuk SETTABLE: R[A][RK(B)] = RK(C)
    Untuk CALL: R[A] = function, B-1 = args, C-1 = returns
    Untuk SELF: R[A] = R[B], R[A+1] = RK(C) (method lookup)

  iABx format:
    opcode(6 bit) | A(8 bit) | Bx(18 bit)
    Untuk LOADK: R[A] = K[Bx]
    Untuk CLOSURE: R[A] = closure(proto[Bx])

  iAsBx format:
    opcode(6 bit) | A(8 bit) | sBx(18 bit signed)
    Untuk FORPREP: R[A] -= R[A+2], PC += sBx
    Untuk JMP: PC += sBx

================================================================================
PATCH STRATEGY
================================================================================

STRATEGI: HOOK FOPEN/FREAD + BYTECODE STRING REPLACEMENT

Mengapa memilih hook fopen/fread:

1. fopen dan fread adalah fungsi libc x86_64 ASLI (bukan ARM translation).
   Frida di sisi x86_64 bisa meng-intercept fungsi ini tanpa masalah.
   Ini sudah diverifikasi: fopen/fread hooks BERHASIL fire di session
   sebelumnya (find_battle_deploy.js).

2. Semua fungsi Lua API (lua_pcall, luaL_loadbuffer, dll) tidak bisa
   diakses karena:
   - libagame.so di-link static dengan Lua library
   - Export tidak ada di .dynsym
   - libhoudini ARM translation membuat symbol tidak visible dari x86_64
   - Dobby hook tidak work karena ARM code

3. File .mt dibaca oleh game melalui fopen/fread standar. Dengan hook
   ini kita bisa intercept buffer SEBELUM di-load oleh Lua loader.

Mengapa patch dilakukan di memory:

1. File APK tidak bisa dimodifikasi langsung (signed APK).
2. File di data directory game mungkin di-verify integrity-nya.
3. Patch di memory hanya mempengaruhi runtime instance, tidak merusak
   file asli.
4. Frida memungkinkan WriteByteArray ke memory proses target.

Mengapa tidak memodifikasi file secara langsung:

1. File .mt asli di APK terenkripsi format Antm (Roo).
2. Decryption key/tool tidak tersedia (hanya binary game yang bisa
   decrypt).
3. File hasil decrypt (.mt.luac) di data/game cache hanya ada setelah
   game selesai initialize, dan mungkin di-verify CRC-nya.
4. Patch di memory lebih aman dan mudah diiterasi.

================================================================================
BYTE PATCH
================================================================================

PATCH 1: isHeroNotOwned -> getHeroSelected

ALASAN:

  "isHeroNotOwned" = 14 karakter
  "getHeroSelected" = 14 karakter
  PANJANG SAMA, jadi tidak perlu mengubah size field.

  Di constant table Lua bytecode, string entry memiliki format:
    type(1 byte, 0x04) | size(4 bytes) | content(size bytes, null terminated)

  Sebelum patch:
    04 0F 00 00 00 69 73 48 65 72 6F 4E 6F 74 4F 77 6E 65 64 00
    type=4(STRING), size=15(14 chars+null), content="isHeroNotOwned\0"

  Sesudah patch:
    04 0F 00 00 00 67 65 74 48 65 72 6F 53 65 6C 65 63 74 65 64 00
    type=4(STRING), size=15(14 chars+null), content="getHeroSelected\0"

  Size field = 0x0F = 15 tetap sama karena kedua string 14 karakter + null.

MENGAPA SIZE FIELD TIDAK BERUBAH:

  "isHeroNotOwned" (14 chars) + null(1) = 15 bytes
  "getHeroSelected" (14 chars) + null(1) = 15 bytes
  Size tetap 15. Tidak perlu mengubah size field.

MENGAPA "getHeroSelected":

  getHeroSelected adalah method yang SUDAH ADA di CommonArrangeHelper
  (k12 di main function constants). Method ini digunakan untuk mengambil
  hero yang sedang dipilih.

  Efek perubahan: self:isHeroNotOwned(heroId) berubah menjadi
  self:getHeroSelected(heroId).

  getHeroSelected kemungkinan return:
    - Hero object jika hero tersebut sedang dipilih
    - nil jika hero tidak sedang dipilih

  Nil bersifat "falsy" di Lua. Dalam konteks kode:
    if self:getHeroSelected(heroId) then return end

  Untuk hero yang sudah dipilih: return non-nil (truthy) -> return (skip)
  Untuk hero yang BELUM dipilih: return nil (falsy) -> lanjut (select)

  Untuk hero tak terowned yang BELUM dipilih: return nil -> LANJUT
  -> Hero bisa dipilih meskipun tidak terowned!

ALTERNATIF YANG DI-PERTIMBANGKAN:

  1. "isHeroSelected" (13 chars) - beda panjang, perlu update size field
  2. "isBaseHeroSelected" (18 chars) - beda panjang
  3. Empty string - akan crash (method lookup return nil, call nil error)
  4. NOP conditional jump - terlalu kompleks karena obfuscation

LOKASI PATCH:

  Semua 4 occurrences dari "isHeroNotOwned" di file akan di-patch.
  Termasuk yang di main function (offset 1355). Resiko: method registration
  bisa berubah, tetapi karena getHeroSelected sudah terdaftar dengan
  method yang benar, efeknya minimal.

  Priority patch: occurrence [2] (main/f7) dan [3] (main/f29) yang
  merupakan method call sites.
